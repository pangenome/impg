## Routing diagnosis

I first started the literal requested shape:

```bash
target/release/impg crush \
  -g /home/erikg/impg/data/c4_low_seqwish_k_20260605T140000Z/graphs/c4.k311.poa2kb.gfa \
  -o /home/erikg/impg/data/c4_motif_local_poa_condense_20260605T175801Z/graphs/c4.k311.poa2kb.motif-local-poa3.gfa \
  --method motif-local \
  --max-iterations 3 \
  --motif-max-window-bp 10k \
  --auto-spoa-max-traversal-len 100k \
  --auto-poasta-max-traversal-len 0 \
  --polish-method poa \
  --polish-rounds until-done \
  -t 32 -v 1
```

That was stopped after the logs showed FastGA/SweepGA replacement calls. The cause is method routing, not missing candidates: `src/resolution.rs` has `motif_local_replacement_method`, and for `--method motif-local` it returns POASTA when the POASTA tier is enabled, otherwise SweepGA. It only returns POA when the outer method is explicitly `poa`.

To run the intended fixed global SPOA replacement while keeping motif-local candidate discovery, I used the supported route:

```bash
/usr/bin/time -v target/release/impg crush \
  -g /home/erikg/impg/data/c4_low_seqwish_k_20260605T140000Z/graphs/c4.k311.poa2kb.gfa \
  -o /home/erikg/impg/data/c4_motif_local_poa_condense_20260605T175801Z/graphs/c4.k311.poa2kb.window-motif-poa3.gfa \
  --method poa \
  --window-mode motif \
  --max-iterations 3 \
  --motif-max-window-bp 10k \
  --polish-method poa \
  --polish-rounds until-done \
  -t 32 -v 1
```

This path exercises `build_poa_replacement`, which calls `build_global_spoa_engine`.
